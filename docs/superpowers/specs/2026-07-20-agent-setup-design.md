# Automatic Agent Setup Design

## Goal

Add a single `opsi agent setup` command that discovers compatible installed agent hosts and installs the complete OPSI Agent Skills repertoire into their global skill locations. The flow should feel like Caveman's one-command installer while preserving OPSI's structured errors, safe filesystem behavior, deterministic generated skills, and cross-platform support.

## Scope

This change adds setup only. It does not add hooks, model runtimes, MCP servers, agent-specific rules, a background updater, or uninstall/status subcommands. Re-running setup is the update path: the installer replaces the known OPSI skills with the repertoire generated by the currently installed `opsi` version.

The public command is:

```sh
opsi agent setup
```

Supported options are:

- `--agent <ids...>` to target explicit installer agent IDs instead of discovery.
- `--all` to target every installer profile and skip prompts.
- `--copy` to copy skill files when symlinks are unavailable or unwanted.
- `--yes` to select all detected agents without confirmation.
- `--dry-run` to return the planned setup without creating temporary files or invoking the installer.

Setup is global by design. A machine-level agent installer should not add `.agents/`, agent-specific directories, or `skills-lock.json` to the user's current project. Project-scoped installation remains available through the documented `npx skills add https://github.com/0xfa7ca7/opsi` workflow.

## Approaches considered

### 1. Maintain an OPSI provider registry

OPSI could own every agent's detection probes and global skill path. This would avoid a runtime dependency, but it would duplicate a fast-changing registry of dozens of agents and make OPSI responsible for platform-specific symlink, copy, XDG, extension, and legacy-path behavior.

### 2. Shell out to `npx skills@latest`

This closely matches Caveman's current provider mechanism and minimizes code, but it downloads executable code during setup and allows behavior to change independently of the installed OPSI version.

### 3. Use a pinned `skills` runtime dependency

This is the selected approach. OPSI depends on an exact `skills` package version and invokes its installed Node entrypoint with an argv array. The dependency owns agent detection and installation paths; OPSI owns the generated source, command contract, dry-run behavior, typed errors, and version upgrades. There is no shell, no `npx`, and no GitHub fetch during setup.

## Architecture

### Installer adapter

Create `apps/cli/src/agent-setup.ts` with a small adapter boundary:

```ts
export interface AgentSetupRequest {
  readonly agents?: readonly string[];
  readonly all?: boolean;
  readonly copy?: boolean;
  readonly yes?: boolean;
  readonly dryRun?: boolean;
}

export interface AgentSetupResult {
  readonly installer: string;
  readonly scope: "global";
  readonly selection: "detected" | "all" | readonly string[];
  readonly skills: readonly string[];
  readonly dryRun: boolean;
}

export interface AgentInstallerRunner {
  run(request: AgentInstallerRunRequest): Promise<number>;
}
```

The production runner resolves the pinned `skills/bin/cli.mjs` entrypoint and executes it with `process.execPath` and an argv array. It never invokes a shell. The runner receives the current working directory, environment, stdio mode, generated source directory, and exact options so tests can replace process execution with a deterministic fake.

### Generated local source

For a real setup, the adapter creates a mode-`0700` temporary directory, calls the existing `generateAgentSkills` renderer into that directory, invokes:

```text
node <resolved-skills-cli> add <temporary-source> --global --skill * [selection flags]
```

and removes the temporary directory in a `finally` block. The local source causes the universal installer to skip remote cloning and install telemetry. The generated contents therefore exactly match the installed `opsi` version.

`--dry-run` validates options and returns a structured plan without creating the temporary source, resolving the installer binary, or starting a child process.

### Interaction modes

With no selection flags, the universal installer detects installed agents. One detected agent is selected automatically; multiple detected agents use its normal selection prompt; no detected agents use its normal supported-agent prompt.

`--yes` keeps discovery but selects all detected agents without prompting. `--agent` accepts one or more explicit installer IDs. `--all` maps to the installer's explicit all-agent mode and implies non-interactive confirmation. `--all` conflicts with `--agent`; duplicate agent IDs are rejected before process execution.

Human output keeps the child installer attached to the configured stdin, stdout, and stderr so prompts work. Structured OPSI output (`--json`, `--ndjson`, `--csv`, `--tsv`, or `--output-format`) requires `--yes`, `--agent`, or `--all`; the runner captures child output and OPSI emits only its normal structured result envelope. This prevents installer decoration from corrupting machine-readable stdout.

### Command integration

Add `agent setup` to the command manifest and register it through `apps/cli/src/commands/agent.ts`. Extend `ProgramDependencies` with an optional `agentInstallerRunner` for tests. The command belongs to the `opsi-diagnostics` Agent Skill so the existing exact command-ownership validator continues to cover the full CLI surface.

## Errors and safety

- Conflicting or duplicate selection options fail before temporary files or child execution with exit category 2. Unknown agent IDs are rejected by the pinned installer and reported as setup failures with its diagnostic preserved.
- Missing or unresolvable pinned installer code returns `AGENT_INSTALLER_UNAVAILABLE` with exit category 5.
- A nonzero installer exit returns `AGENT_SETUP_FAILED`; invalid installer input maps to exit category 2 when identifiable, otherwise internal failure maps to exit category 1.
- Temporary source creation or cleanup failures use the existing skill-generation error model without exposing secret environment values.
- Installer execution uses `process.execPath` and an argv array, never a shell command string.
- Only the generated temporary source is deleted. Agent destinations are owned and updated by the pinned installer.
- Dry-run performs no filesystem or child-process mutation.

## Documentation and release contract

Update the root and packaged CLI READMEs, command reference, and Agent Skills index to make `opsi agent setup` the recommended machine-wide setup command. Retain `npx skills add` as the project-scoped and advanced fallback. Document explicit-agent, all-agent, copy, yes, dry-run, and structured-output behavior.

Add a minor changeset for `opsi`. Regenerate the checked-in skill tree so `opsi-diagnostics` documents `agent setup` and all manifest commands remain owned exactly once.

## Verification

Tests cover:

- exact command manifest flags and completion surface;
- option validation and argument construction without shell interpolation;
- default detection, explicit agents, all agents, copy, yes, and conflicts;
- dry-run producing no temporary directory or runner call;
- generated source contents and guaranteed cleanup on success and failure;
- typed missing-installer and child-process failures;
- structured-mode non-interactive requirements and clean output;
- byte-for-byte checked-in skill drift;
- packed npm installation resolving the pinned installer and running `agent setup --dry-run --json`;
- full `pnpm check` on the final branch.
